## 4/9晚上 — Review代码，发现不对

Phase 9（Discovery Refactor）刚做完。29个plan全部执行完毕，201次commit。从数字上看，这个项目进展非常充分。

但晚上Review代码的时候，越看越不对。

Detent最初的愿景很清楚：**基于控制论的pipeline编排**。核心是约束传播——需求变成约束，约束冻结后指导agent行为，agent产出被评估，不达标则回退重来。这是一个控制系统的闭环。

但实际的Roadmap是这么编排的：

| 里程碑 | Phase | 内容 |
|--------|-------|------|
| M1 Engine | 1-2 | CLI工具 + Pipeline骨架 |
| M2 Agents | 3-5 | 真值面 + 对抗规划 + 编码循环 + 可观测性 |
| **M3 Web UI** | **6** | **可视化看板** |
| （无里程碑） | 7 | 控制论重构 |
| （无里程碑） | 8 | NPX打包 |
| （无里程碑） | 9 | Discovery重构 |

看到问题了吗？**Web UI有自己的里程碑（M3），控制论重构没有。**

Phase 6（Web UI）一个phase就消耗了9个plan——占全部29个plan的31%。而且Phase 6做出来之后，Blog 8-2里已经发现了：Claude Code的headless模式不能被HTTP触发，WebUI能做的只有监控，不能做控制。一个占了三分之一工作量的feature，最终被砍掉了一半的功能。

与此同时，**控制论pipeline编排——这个项目的核心理念——直到Phase 7才被认真对待**。不在原始Milestone里。是后面"插入"的。

这说明什么？**Roadmap本身就编码了错误的优先级。**

---

## 为什么会偏

回头看，偏离不是某一天突然发生的，是一步步走过来的。

**Phase 1-2**做CLI和pipeline骨架，合理。

**Phase 3-5**做真值面和agent模板，也合理。但这时候已经有一个隐患：真值面的设计是"所有agent都能写入"，Blog 8-2后来发现这是个根本性的架构错误（concurrent write conflict）。但当时觉得"先跑起来，后面再fix"。

**Phase 6是转折点。** 原始设计里WebUI是M3里程碑——一个完整的可视化控制面板，能启动pipeline、配置参数、实时监控。这个需求本身很诱人：做出来之后可以demo、可以截图、用户能直观感受到框架在干什么。AI也特别擅长做这类有明确UI mockup的任务——给它一个Svelte + Fastify + WebSocket的技术栈，它能很快地把页面搭出来，每个plan都能看到具体的产出。9个plan下来，确实有了一个完整的前端。

但做这些的时候，pipeline的控制论核心还停留在Phase 3的"每个agent自己写真值面"的初始版本。

**Phase 7才开始补核心。** parent-as-sole-writer、4状态约束生命周期、Discovery的发散-收敛链路——这些都是这个时候才做的。如果这些是框架的灵魂，它们应该在Phase 2-3就出现，不是Phase 7。

Phase 8做NPX打包，Phase 9做Discovery重构。每个phase都是在修前面遗留的结构性问题。到Phase 9做完，代码里到处是后期插入的修补——Phase 7的控制论重构要兼容Phase 3-5的老结构，Phase 8的打包要处理Phase 6 WebUI和Phase 5可观测性模块的路径问题。

整个项目的结构变得散乱。

## Claude的上下文在失焦

项目膨胀还带来了一个实操层面的问题：**Claude对codebase的全局感知在下降。**

9个phase、29个plan执行完之后，项目里有agents/、bin/、commands/、engine/、web/、test/六大目录，加上.planning/下的规划文件、.claude/下的配置。每次开新session，Claude需要重新理解这个越来越大的codebase。

我注意到后期的session里，Claude在做修改时经常忽略其他模块的关联影响。改了engine/里的状态管理逻辑，没同步commands/里对应的skill。加了一个新的CLI命令，没更新web/里的API route。不是它犯了"错误"，而是在codebase膨胀到一定程度后，它没有足够的上下文窗口来同时关注所有相关文件。

这其实是Blog 3里"Lost in the Middle"问题（[Liu et al., 2023](https://arxiv.org/abs/2307.03172)）的另一种表现。那次是单个prompt里的sibling context太长，这次是整个项目的codebase太散太大，模型的注意力在跨模块关联上衰减。问题的本质是一样的：**上下文规模超过模型的有效处理范围后，局部最优的修改不等于全局一致的修改。**

Blog 5提炼的"session上下文管理是AI Coding的第一质量杠杆"（认知13），在这里得到了项目级别的验证：不只是单次session的上下文要管，**项目整体的上下文负载**也要管。codebase越膨胀，每次session里Claude能真正"看到"的比例就越低，产出一致性就越差。

---

## 理念偏离是怎么累积的

还有一个更深层的问题。

Detent的核心理念是"控制论pipeline编排"。但在实际执行中，这个理念不断被其他需求稀释：

- Phase 3做了真值面，但写入模型是错的（所有agent都能写）。当时想"先跑起来"
- Phase 5加了可观测性，引入了Gemini CLI做行为日志——一个独立的子系统。核心pipeline还没完善，就开始建周边
- Phase 6做了WebUI，9个plan。核心pipeline更远了
- Phase 7才回来补控制论核心，但这时候要兼容前面5个phase的结构
- Phase 8做打包，Phase 9做Discovery重构——都在修补

每一步的偏离看起来都不大。Phase 5的可观测性"挺重要的"，Phase 6的WebUI"让用户体验更好"。每个理由单独看都成立。但累积起来，核心理念在roadmap里被不断往后推。

这个模式在产品管理里有个名字叫 **feature creep**——不是某个大决定出了问题，而是一系列看起来合理的小决定，累积起来把产品带偏了（[Brooks, 1975](https://en.wikipedia.org/wiki/The_Mythical_Man-Month)，讲的是软件项目里"第二系统效应"的类似现象）。

AI在这个过程中是完美的放大器。你说"做WebUI"，它不会说"你确定？核心pipeline还没完善"。它会立刻开始设计组件、选技术栈、拆plan。每个plan的产出都很完整，进度条每天都在动。**feature creep在AI Coding里更隐蔽，因为AI让每个偏离都看起来很"完成"。**

## 决定弃置

4/10早上，把Detent和最初的ECL构想放在一起对比。

ECL的核心是一条10阶段的控制论pipeline：需求→澄清→发散→收敛→挑刺→红蓝对抗→审查→编译→编码→评估。每个阶段的产出受约束传播控制，真值面是唯一的信息中枢。

Detent做到了什么？CLI工具有了，agent模板有了，WebUI有了，NPX打包有了。但这些都是**脚手架**。核心的控制论编排——约束怎么传播、阶段之间怎么门控、真值面怎么驱动agent行为——做得晚、做得散、而且被前面的架构决策约束住了。

一个直观的对比：

**Detent（201 commits）：** agents/ + bin/ + commands/ + engine/ + web/ + test/，六大目录，Phase 6 WebUI占31%工作量，控制论核心是Phase 7后期插入

**bonfire（新项目）：** bin/lib/（核心逻辑）+ schemas/ + templates/ + references/ + tests/，全部围绕ECL pipeline，没有Web UI，没有可观测性子系统，没有打包逻辑——先把核心做对

bonfire用Claude Code原生原语（skills、agents、hooks、Node.js CLI）重新实现ECL pipeline。不是在Detent上继续修补，是从头开始，这次先做核心。

弃置Detent不是因为201次commit白费了——Blog 4已经说过，**补丁式修复解决不了架构问题，要敢停下来重构**（认知9）。那次是mosaicat的QA架构，这次是Detent的整体优先级。只不过这次的代价更大——不是重构一个模块，是弃置一个做了5天的项目。

但Blog 4也说了另一件事：**应该更早停下来**。3/22就感觉不对，多拖了两天。这次也一样。Phase 6做WebUI的时候就隐约觉得"是不是应该先做核心"，但9个plan的产出很丰富，"快做完了"的感觉太强，又多做了3个phase。

Blog 4的认知10：**AI不会主动质疑你的方向，批判性思考得自己来。** 这次再一次验证了。

## 这几天做得好的 / 做得不好的

**做得好的：**
- 最终停下来了。虽然201 commits的沉没成本很大，但及时止损比继续在错误方向上投入更重要
- 弃置不是冲动决定，是基于具体的代码review和原始构想对比做出的
- bonfire一开始就把控制论核心放在最前面——汲取了Detent的教训

**做得不好的：**
- Roadmap设计时就没把核心放在核心位置。WebUI拿了独立里程碑，控制论重构没有
- Phase 3的真值面写入模型有问题，当时选择了"先跑起来后面fix"，这个决定的代价在Phase 7才完全暴露
- 9个plan做WebUI的时候没有停下来问"核心做好了吗"——被AI的高效产出感麻痹了
- codebase膨胀后没有主动做模块收敛，让Claude的上下文负载越来越大

## 反思

这篇blog和Blog 4写的其实是同一个故事的不同版本。

Blog 4是mosaicat的补丁循环——每天commit很多但问题不收敛，最终停下来做了架构重构。这次是Detent的优先级偏移——每个phase都很完整但核心越来越薄，最终停下来换了个项目。

两次的共同点是：**AI配合得太好了。** 它会把你指向的任何方向做得很完整、很专业。你要WebUI，它给你9个plan的完整前端。你要NPX打包，它给你installer脚本。每一步都有产出，每一步都"完成"了。这种高效的执行力会掩盖方向问题，因为"产出很多"的感觉和"在正确的方向上推进"的感觉太像了。

区分两者靠什么？靠定期回到原始构想去对比。"我当初要做什么？现在做的东西离那个目标有多远？"这个问题AI不会替你问。

产品管理里有一个实践叫 **North Star Review**——定期回到产品的核心指标和愿景，检查当前的feature优先级是否还对齐。在AI Coding里同样需要：**定期回到最初的构想，检查你的roadmap是否还在服务核心目标。** 如果核心功能在roadmap里的位置越来越靠后，那就是最强的偏离信号。

最后一个教训：**"先做完再fix"是技术债，"先做别的再做核心"是方向债。** 技术债有利息但还得起，方向债积累到一定程度就只能破产重来。Detent的201次commit就是这个方向债的利息。
